# チーム開発演習(Git/GitHub)　最終課題　レベル3
## 概要
このリポジトリのWebサイトを、自分のオリジナルサイトに修正する
- 既存のコードを解析し、ページの構造やBootstrapの使われ方を理解した上で、自分のオリジナルサイトへ内容を変更する  
- コードの構造や記述方法を把握しながら、整理・改善（リファクタリング的作業）を行う  
  - ページ遷移、CSSやJSの整理など、全く手を付けていないWebサイトなので、cloneするファイルをテンプレートだと考えて修正をすること

## このリポジトリのWebサイトについて
HTMLファイル7つで構成されています。  
Webサイトとして機能させるためには、多くの箇所を修正する必要があります。  
※レベル1とレイアウトが結構異なるので、レベル1を参考にできる部分はありますが、ほぼ自分で1から解読して解析し、修正しなければならないです

## ルール
- 既存のページ構成・大まかなレイアウトは崩さず、オリジナルのWebサイトに書き換えること(`index.html`以外のファイル名の変更はしていいが、ページの削除はしないこと)
  - **Webサイトとして機能させること(リンク先がない、等はNG)**
- 全体で最低2か所はレイアウトの一部を変更すること(Bootstrapを活用)
    - 色変更ではなく、配置や見え方を工夫すること
    - どこを変更したか、を発表時に聞きますので、「忘れました」や「AIに作ってもらって解析していないので理解していません」ということの無いように
- リンクが記述してあるのに遷移先が存在しないページ(新規登録ページなど)を、少なくとも2ページ追加すること(他のページとデザインのレイアウトを合わせること)
  - 元のHTMLファイルが7つなので、9ページは最低用意する
  - 9ページより多くリンクの種類が貼られているので、用意している9ページ分はきちんとパスを通し、それ以外の用意していないページについては、自分自身に遷移させる、でOK
- 画像はダミー画像でも構いませんが、何が表示されるかが分かるダミー画像を用意すること
- レスポンシブレイアウトを適用すること
  - 崩れているレイアウトがないか、しっかりと確認を
- 色・フォントは自由に変更可
- ファビコンをWebサイトの内容に合うよう変更すること
- クローン元のファイルでバグやレイアウト崩れを発見したら修正すること
- コードの可読性を意識すること
    - インデントを揃える
    - **意味のあるクラス名を付ける**
    - 共通するCSS、JSは別ファイルへ記述し、HTMLファイルへの記述をなるべく減らすこと
- 使われていないCSSなど、不要なコードを発見したら削除してOK
- Webサイトの概要が分かるREADME.mdを配置すること

## 補足
Issueを立てたり、タスク管理のツールでやらなければならないことを管理しておくと、抜け漏れがなく効率的に作業できると思います。  
また、README.mdに、使用したBootstrapコンポーネントや工夫した点、オリジナル要素をまとめておくと、発表時に説明しやすいかと思われます。

## 手順
1. 任意の場所にcloneする
2. リモートリポジトリをpublicで作成し、cloneしたリポジトリの接続先を、作成したリモートリポジトリに変更する
    ```bash
    git remote set-url origin 自分のリポジトリURL
    ```
3. 作業開始    
    適宜ブランチを切って、作業単位ごとにcommitするなど工夫して作業してください。  
    【ブランチ運用のメリット】
    - 作業内容を分けて管理できる
    - 誤ってmainを壊すリスクを減らせる
4. 作業終了後は、自分のリモートリポジトリにpushする
5. GitHub Pagesで公開する  
    [GitHub PagesでのWebサイト公開手順](https://prog-8.com/docs/github-pages)  
    上記手順の「5 ページの公開」を参考にしてください。
6. Classroomに公開したURLを提出
